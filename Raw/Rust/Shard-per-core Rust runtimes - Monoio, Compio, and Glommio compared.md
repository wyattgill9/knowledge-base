
**Three Rust async runtimes — Monoio, Compio, and Glommio — reject Tokio's work-stealing model in favor of shard-per-core execution, each making fundamentally different architectural bets.** Monoio optimizes for raw network throughput via zero-copy buffer ownership, Compio prioritizes cross-platform flexibility with a pluggable driver-executor architecture, and Glommio delivers the most sophisticated scheduler and storage I/O subsystem in the Rust ecosystem. All three achieve **2–3× Tokio's throughput at 16+ cores** by eliminating cache coherency traffic and atomic contention, but they pay for this with queueing-theory penalties, ecosystem fragmentation, and cooperative scheduling risks. The shard-per-core model traces directly to ScyllaDB's Seastar framework, and its adoption in Rust represents both a significant performance opportunity and a substantial engineering commitment.

## Why shard-per-core exists and what it actually fixes

The shard-per-core paradigm solves a hardware problem that software cannot abstract away: **cache coherency is the dominant cost in multi-threaded systems at scale**. When Tokio's work-stealing scheduler migrates a task from core A to core B, every cache line that task touches must be invalidated on A and reloaded on B — a process governed by the MESI protocol that costs **50–200 nanoseconds per cache line** under contention. False sharing, where independent data happens to occupy the same 64-byte cache line, can degrade throughput by 10× or more. Even uncontended atomic operations (the backbone of lock-free work-stealing queues) carry a 5× penalty over non-atomic equivalents due to LOCK prefix instructions forcing memory ordering.

The Seastar framework, built by Avi Kivity's team at ScyllaDB, proved the alternative: pin one thread per physical core, give each thread its own memory allocator, network queue, and storage partition, and communicate only through explicit message passing. ScyllaDB claims near-linear scaling — doubling cores doubles throughput — because there is essentially no Amdahl's Law serial bottleneck from inter-core coordination. Glauber Costa brought this philosophy from seven years at ScyllaDB directly into Glommio; ByteDance independently arrived at the same conclusions building Monoio for their proxy infrastructure; and Compio's author recognized the need for a cross-platform completion-based runtime.

The io_uring kernel interface amplifies the shard-per-core advantage. Traditional epoll-based runtimes like Tokio use readiness notification: the kernel tells userspace a socket is readable, then userspace issues a `read()` syscall. With io_uring, userspace submits I/O requests to a Submission Queue and harvests results from a Completion Queue — both lock-free ring buffers in shared memory. **Multiple operations batch into a single `io_uring_enter` syscall**, and checking for completions requires only comparing two integers (head and tail pointers), with no syscall at all. For file I/O, the difference is even starker: Tokio considers files "always ready" under epoll and delegates to a blocking thread pool of up to 512 threads, while io_uring performs truly asynchronous file operations.

The fundamental trade-off is captured by queueing theory. Josh Snyder's 2024 analysis quantifies it precisely: at 960K RPS with 100μs average service time and a 50% latency tolerance, a shared queue (M/M/k) needs **98 CPUs**, while shard-per-core (independent M/M/1 queues) needs **288 CPUs** — a **3× overprovisioning penalty**. Each shard saturates independently, and there is no mechanism to redistribute excess capacity from idle shards to hot ones. This is the price of eliminating coordination.

## Monoio: ByteDance's zero-copy network engine

Monoio is ByteDance's thread-per-core runtime, open-sourced through CloudWeGo and deployed in production powering their Monolake proxy framework. Its defining architectural decision is the **ownership-transfer buffer model**, which enables true zero-copy I/O at the cost of ecosystem compatibility.

Each CPU core runs an independent `Runtime<D>` instance with its own event loop, task queue (`VecDeque`), and io_uring ring. Tasks are `Rc<Task>` (not `Arc`), wakers manually manage reference counts via `Rc::into_raw`/`Rc::from_raw`, and spawned futures need not be `Send` or `Sync`. Multi-core deployment uses `SO_REUSEPORT` so each thread's listener accepts connections independently — the kernel distributes incoming connections, eliminating cross-thread work distribution entirely.

The `AsyncReadRent` and `AsyncWriteRent` traits are Monoio's most consequential design choice. Where Tokio's `AsyncRead::poll_read` borrows a `&mut [u8]`, Monoio's `read<T: IoBufMut>(buf: T) -> BufResult<usize, T>` takes ownership of the buffer and returns it alongside the result. This exists because **io_uring requires stable buffer addresses** — the kernel holds a pointer to the buffer during the asynchronous operation, and if Rust's ownership system allowed the buffer to be moved or dropped, the result would be a use-after-free in kernel space. By consuming the buffer, Monoio guarantees pointer stability without runtime checks. The `IoBuf` and `IoBufMut` traits provide `read_ptr()` and `write_ptr()` for safe kernel access.

Monoio tracks in-flight operations via a `Slab<Lifecycle>` indexed by slot number, which maps directly to io_uring's `user_data` field in completion queue entries. This slab-based allocation avoids heap allocation per I/O operation — a concrete advantage over Compio's boxing approach. The driver abstracts across backends through the `OpAble` trait, which requires implementations to provide both `uring_op()` (returning an io_uring SQE) and `legacy_call()` (performing the blocking syscall directly). The `FusionDriver` probes for io_uring support at startup by attempting `IoUring::new(2)` and falls back to mio-based epoll/kqueue transparently.

The scheduler is purely cooperative with **no fairness guarantees**. The main loop drains the task queue in FIFO order, polls the main future, and calls `driver.park()` when no tasks are runnable. There is no budget system analogous to Tokio's `coop` module — a compute-heavy task that never yields will starve all other tasks on that core. Monoio provides `yield_now()` as a manual escape hatch, but discipline falls entirely on the developer.

ByteDance's benchmarks on production hardware show the payoff clearly: at **4 cores, Monoio delivers ~2× Tokio's throughput; at 16 cores, ~3×**. Tokio's throughput plateaus and even degrades at high core counts due to work-stealing contention. However, at very low connection counts on a single core, Monoio's latency is higher than Tokio's because io_uring's ring setup and batched submission model adds overhead that epoll's direct readiness check avoids. File I/O benchmarks from the Tonbo/Fusio project show Monoio completing 4KB random reads in **3.78μs versus Tokio's 110μs** — a 29× improvement attributable to true async I/O versus Tokio's blocking thread pool.

## Compio: cross-platform completion I/O with a pluggable architecture

Compio, authored by Yuyi Wang (Berrysoft), occupies a unique position: it is the only shard-per-core Rust runtime with first-class support for **both io_uring and Windows IOCP**, plus a polling fallback for macOS. Its architecture disaggregates the I/O driver from the executor — a design choice that proved decisive for its most significant adopter, Apache Iggy.

The `compio-driver` crate provides a platform-agnostic `Proactor` abstraction with `push()` (submit operation), `poll()` (wait for completions), `pop()` (retrieve results), and `cancel()`. On Linux, this wraps io_uring with what multiple evaluations describe as **the broadest io_uring feature coverage** among Rust runtimes — buffer pools, personalities, splice operations, statx, and active tracking of new kernel features. On Windows, it wraps IOCP natively, avoiding the undocumented APIs that Tokio's mio layer uses. The Proactor is deliberately `!Send` and `!Sync`, enforcing single-threaded-per-core semantics at the type level.

Compio's most consequential architectural decision is **boxing each I/O request** submitted to the submission queue, incurring a heap allocation per operation. Monoio's slab allocator avoids this entirely. The Compio team made this trade-off consciously: slab allocation would add complexity to the cross-platform executor, particularly for the IOCP backend where submission semantics differ from io_uring. Apache Iggy's team found the overhead negligible when using mimalloc, confirming that for small, predictable allocations the cost is dominated by I/O latency.

The driver-executor separation is Compio's key differentiator. Users can build custom executors while reusing Compio's battle-tested I/O driver. Apache Iggy's team highlighted this as critical for **Deterministic Simulation Testing (DST)** — the ability to replace the time wheel, scheduler, and driver with deterministic mocks for reproducible testing of distributed system behaviors. Neither Monoio nor Glommio supports this architectural pattern; their drivers and executors are tightly coupled.

Apache Iggy's migration from Tokio to Compio (completed December 2025) represents the most thoroughly documented production adoption of any shard-per-core Rust runtime. The results: **5,000 MB/s throughput** (5 million messages per second at 1KB each), sub-millisecond P99 latency, and ~9.5ms P9999 WebSocket latency with fsync-per-message persistence. The Iggy team chose Compio over Monoio (citing slower maintenance and io_uring feature gaps) and Glommio (citing effective abandonment and opinionated design disagreements). They reported patches merged "within hours" — a maintenance velocity neither competitor matches.

Compio's buffer ownership model mirrors Monoio's approach: `AsyncRead` takes ownership via `read<B: IoBufMut>(buf: B) -> BufResult<usize, B>`. This creates the same ecosystem incompatibility. When Iggy needed WebSocket support, they discovered `async-tungstenite` was fundamentally coupled to poll-based I/O and had to create `compio-ws` from scratch. The broader incompatibility with Tokio's ecosystem — reqwest, axum, tonic, tower — remains the most significant practical barrier.

## Glommio: Seastar's spiritual successor for storage workloads

Glommio is the most architecturally ambitious of the three, designed by Glauber Costa at DataDog after seven years building ScyllaDB's Seastar framework. It features a triple io_uring ring design, proportional-share scheduling with latency classes, and Direct I/O as a first-class citizen — capabilities no other Rust runtime offers. It is also **effectively unmaintained** since Costa's departure to co-found Turso in 2021.

The triple-ring architecture is Glommio's most distinctive innovation. Each thread operates three independent io_uring instances: a **Main Ring** for throughput-oriented operations, a **Latency Ring** for time-sensitive tasks where a timer ensures yield within a specified duration, and a **Poll Ring** for NVMe operations using io_uring's interrupt-free polling mode. Before the main ring blocks, the latency ring's file descriptor is registered for poll on the main ring, ensuring latency-sensitive completions wake sleeping threads. Checking the latency ring for pending events requires only comparing two integers (CQ head and tail pointers) — making `yield_if_needed()` nearly free.

The scheduler implements **proportional-share fairness** through task queues with configurable `Shares`. If queue A has 2 shares and queue B has 1 share, A receives 2/3 of CPU time under contention. Shares can be static or dynamic, adjusted at runtime by **embedded controllers** that observe metrics and automatically tune scheduling weights — a feedback loop for building self-tuning storage systems. Each queue also declares a `Latency` class: `NotImportant` (throughput-optimized, longer time slices) or `Matters(Duration)` (latency-sensitive, with io_uring timer-driven preemption). The default preemption timeslice is **100ms**, configurable per executor. A stall detection handler sends `SIGUSR1` and can collect stack traces when tasks exceed their expected runtime — observability that neither Monoio nor Compio provides.

Glommio's storage I/O subsystem is unmatched. `DmaFile` provides random-access Direct I/O that bypasses the OS page cache entirely, allocating from io_uring's pre-registered buffer pool and returning reference-counted pointers for zero-copy device-to-user data transfer. `DmaStreamReader` implements intelligent read-ahead that dispatches new reads as each buffer is consumed, maintaining constant I/O parallelism — unlike OS read-ahead which issues a large batch then waits. Costa's benchmarks on Intel Optane demonstrate the impact: sequential scans of a 53GB file (2.5× RAM) achieved **7.29 GB/s with optimized Direct I/O versus 0.95 GB/s with buffered I/O** — a 7.7× improvement. Random reads hit **547K IOPS** versus 238K IOPS for buffered I/O, using 80KB of memory versus gigabytes of page cache.

However, Glommio pays a performance tax for API familiarity. Unlike Monoio's ownership-transfer model, Glommio copies data into internal managed buffers to provide a more traditional borrowing interface. ByteDance's benchmarks attribute Monoio's higher peak network throughput directly to this difference — Glommio's extra copy adds measurable overhead in network-bound workloads.

## The hidden costs and failure modes

All three runtimes share failure modes inherent to the shard-per-core model that benchmarks rarely capture. **Head-of-line blocking** is the most immediate: a single compute-heavy task that fails to yield blocks all I/O processing on its core. Glommio's stall detector can identify this; Monoio and Compio offer no detection mechanism. **Workload imbalance** is the most systemic: if data access patterns are skewed (hot partitions, popular users), some cores saturate while others idle, and there is no mechanism to redistribute. Monoio's documentation explicitly acknowledges this: "If the workload is very unbalanced, it may cause performance degradation than Tokio since CPU cores may not be fully utilized."

A critical safety issue affects all io_uring runtimes: **future cancellation semantics break under io_uring**. With epoll, dropping a future is safe because no kernel operation is in flight. With io_uring, the kernel holds a pointer to the user's buffer during the async operation. Dropping the future without cancelling the io_uring operation causes use-after-free in kernel space. The Tonbo.io team demonstrated that using `select!` for timeouts causes TCP connection leaks across Monoio, Glommio, and Compio. Monoio provides "cancellable I/O" as mitigation, but cancellation safety is contagious through nested futures with no compile-time enforcement.

The `RefCell` footgun discovered by Apache Iggy is another non-obvious trap. Thread-per-core eliminates the need for `Mutex`, tempting developers to use `RefCell`. But holding a `RefCell` borrow across an `.await` point allows the executor to yield to another future that attempts to borrow the same cell, causing a runtime panic. Iggy's solution was adopting Entity Component System patterns with Struct-of-Arrays decomposition to separate state from storage operations — significant architectural complexity driven by a runtime model constraint.

**Container deployment presents a practical blocker**: Docker, containerd, and other major container runtimes now block io_uring by default in their seccomp profiles following ARMO researchers' demonstration of a rootkit operating entirely through io_uring syscalls that bypassed all major runtime security tools. Any containerized deployment of these runtimes requires custom seccomp profiles — an operational burden that may be unacceptable in enterprise environments.

## When each runtime fits — and when it doesn't

**Monoio excels at high-throughput network proxies and gateways** where workload distribution is naturally even (connections spread uniformly across cores via SO_REUSEPORT) and raw throughput per core matters most. ByteDance's Monolake framework validates this — their gateway outperformed NGINX by 20% and their RPC layer showed 26% improvement over Tokio-based equivalents. Monoio is a poor fit for storage-intensive workloads (lacking Glommio's DMA abstractions), CPU-bound processing (no preemption or fairness), or applications requiring Windows support.

**Compio is the pragmatic choice for new io_uring projects**, particularly those requiring cross-platform deployment, architectural flexibility for testing, or simply active maintenance. Apache Iggy's endorsement carries weight: a production message streaming system achieving 5M messages/second chose Compio specifically for its pluggable driver, responsive maintenance, and broad io_uring coverage. Compio is a poor fit for workloads requiring Glommio-level scheduler sophistication or where the per-operation heap allocation (however small) is unacceptable in extreme hot paths.

**Glommio is purpose-built for storage engines on NVMe hardware**, and nothing else in the Rust ecosystem matches its Direct I/O capabilities, triple-ring architecture, or proportional-share scheduling. The GMF framework demonstrated 84% more requests/second than Tonic-based gRPC. However, Glommio's maintenance trajectory is concerning — the creator left DataDog, open issues accumulate slowly, and a memory corruption bug in the SPSC queue was reported in December 2025. Teams considering Glommio should be prepared to fork and maintain it.

**None of these runtimes should be chosen for general-purpose applications.** The ecosystem incompatibility with Tokio — the inability to use reqwest, axum, tonic, tower, and hundreds of other crates without compatibility shims that negate performance gains — is a practical barrier that no amount of raw throughput overcomes. For teams wanting shard-per-core benefits without ecosystem abandonment, running N independent `tokio::runtime::Builder::new_current_thread()` instances with SO_REUSEPORT captures **1.5–2× improvement** over Tokio's multi-threaded default (validated by TechEmpower's top Rust entries) while retaining full ecosystem access.

## Decision framework and opinionated recommendations

The decision tree is straightforward. First ask: **does your workload naturally shard?** If connections, keys, or partitions don't map cleanly to cores, work-stealing will outperform shard-per-core due to queueing theory alone (3× overprovisioning penalty for M/M/1 versus M/M/k). Second: **are you on Linux with io_uring support and not in locked-down containers?** If not, only Compio offers a path forward, and with reduced benefits. Third: **can you afford to abandon the Tokio ecosystem?** If your service depends on Tokio-native libraries, the migration cost likely exceeds the performance gain.

For teams that pass all three gates, **Compio is the recommended default in 2026**. It is the only actively maintained option with responsive leadership, the only one supporting Windows and macOS, and the only one with driver-executor separation enabling deterministic simulation testing. Apache Iggy's production validation at scale provides concrete evidence of viability.

Choose Monoio over Compio only if you are building a Linux-only network proxy where the slab allocator's zero-allocation I/O submission provides measurable benefit in your specific hot path, and you are comfortable with ByteDance's slower maintenance cadence. Choose Glommio only if you are building a storage engine that specifically needs proportional-share scheduling, DMA file abstractions, and latency-class differentiation — and you have the engineering capacity to maintain a fork.

For everyone else — and this is the majority of Rust async developers — the "poor person's thread-per-core" pattern of running Tokio's `current_thread` runtime per core with SO_REUSEPORT delivers most of the scaling benefit with none of the ecosystem cost. The shard-per-core Rust ecosystem remains fragile: Glommio is fading, Monoio is slowing, and Compio, while thriving, serves a community orders of magnitude smaller than Tokio's. The performance is real, but so is the engineering tax.
